# وثيقة المتطلبات

## 1. نظرة عامة على التطبيق

### 1.1 اسم التطبيق
Vodafone Fakka Premium

### 1.2 المالك
Nader Akram

### 1.3 الإصدار
v3.0.38

### 1.4 الوصف
منصة ويب احترافية لشحن كروت الفكة مع نظام اشتراكات وأكواد ترخيص، مصممة بواجهة عربية فاخرة وتدعم التكامل المستقبلي مع محرك شحن خارجي. تتضمن مركز شبكات متعدد لعرض خدمات Vodafone وOrange وEtisalat وWE، مع نظام إدارة كروت ديناميكي كامل وتحسينات أداء شاملة. تشمل المرحلة الجديدة إعادة بناء لوحة الإدارة بالكامل بصفحات كاملة احترافية، بالإضافة إلى نظام تقييم الكروت واقتراح التعديلات. **يتضمن التحديث الحالي تطبيق 15 طبقة حماية أمنية متقدمة (Zero Trust Architecture) لمنع أي APK معدل أو معاد توقيعه من العمل.**

### 1.5 المبادئ الأساسية
- جميع البيانات ديناميكية (الكروت، الأسعار، الوحدات، الحالات)
- لا توجد بيانات ثابتة أو افتراضية
- النظام جاهز للربط مع API خارجي أو Recharge Engine
- تصميم Premium حديث بنمط Dark Luxury UI
- دعم كامل للغة العربية RTL
- Mobile First و Responsive
- Offline-First مع مزامنة ذكية
- منع التكرار في جميع العمليات
- لوحة إدارة احترافية بصفحات كاملة بدون Popup/Modal
- **Zero Trust Architecture: عدم الثقة بأي بيانات من العميل، جميع القرارات الأمنية من السيرفر**

## 2. المستخدمون وسيناريوهات الاستخدام

### 2.1 المستخدمون المستهدفون
- المستخدمون النهائيون: أفراد يحتاجون لشحن كروت الفكة واستعراض خدمات الشبكات
- المسؤولون (Admin): إدارة المستخدمين والاشتراكات والأكواد والكروت والإصدارات والميزات
- المسؤولون الرئيسيون (Super Admin): صلاحيات كاملة على النظام

### 2.2 السيناريوهات الأساسية
- تفعيل الاشتراك باستخدام كود تفعيل
- شحن كروت الفكة بشكل متكرر
- استعراض خدمات الشبكات المختلفة
- حفظ الأرقام المفضلة للشحن السريع
- مراجعة سجل العمليات السابقة
- مراجعة الإحصائيات التفصيلية
- تقييم الكروت واقتراح تعديلات على بياناتها
- إدارة الاشتراكات والأكواد والكروت (للمسؤولين)
- إدارة الأصول المرئية الديناميكية (للمسؤولين)
- حذف المستخدمين نهائياً (للمسؤولين)
- مراقبة المستخدمين والإصدارات في الوقت الفعلي (للمسؤولين)
- إدارة الميزات والإصدارات بدون تحديث APK (للمسؤولين)
- تنفيذ إجراءات متقدمة على حسابات المستخدمين (للمسؤولين)
- مراجعة تقييمات واقتراحات المستخدمين على الكروت (للمسؤولين)
- **مراقبة المحاولات المشبوهة والحظر التلقائي (للمسؤولين)**

### 3.16 لوحة الأدمن (Admin Dashboard)

#### 3.16.1 Users

##### 3.16.1.1 قائمة المستخدمين (/admin/users)
- عرض قائمة المستخدمين
- بحث وفلترة المستخدمين
- Sorting حسب: تاريخ التسجيل، آخر نشاط، حالة الاشتراك
- Pagination
- عرض معلومات مختصرة لكل مستخدم: الصورة، الاسم، User ID، الحالة، نوع الحساب، آخر نشاط
- **عرض حالة الأمان: آمن، مشبوه، محظور**
- زر للانتقال إلى صفحة تفاصيل المستخدم

##### 3.16.1.2 صفحة تفاصيل المستخدم (/admin/users/{id})
- Header واضح يحتوي على: اسم المستخدم، User ID، زر العودة
- صورة المستخدم
- الحالة: نشط، معلق، محظور
- نوع الحساب: User، Admin، Super Admin
- الإصدار الحالي للمستخدم
- آخر إصدار متاح
- آخر Login
- آخر Activity
- **معلومات الأمان: Device ID، Hardware Profile، Security Token، APK Signature، Play Integrity Status**
- معلومات الاشتراك: حالة الاشتراك، تاريخ التفعيل، تاريخ الانتهاء، الأيام المتبقية، الكود المستخدم
- الإحصائيات: إجمالي العمليات، إجمالي الكروت، إجمالي المبالغ، عدد الأرقام المفضلة
- العمليات الأخيرة (آخر 5 عمليات)
- سجل النشاط (آخر 10 أنشطة)
- سجل التعديلات (جميع التعديلات التي تمت على الحساب)
- سجل الاشتراك (تاريخ جميع الاشتراكات والتجديدات)
- سجل الحظر (إذا تم حظر الحساب سابقاً)
- سجل الأكواد (جميع الأكواد المستخدمة)
- **سجل المحاولات المشبوهة (إن وجدت)**
- سجل كامل لأي إجراء تم على المستخدم
- أزرار Copy بجانب: اسم المستخدم، User ID، رقم الهاتف، البريد، Device ID
- أزرار سريعة: عرض الإجراءات، إدارة الاشتراك، تعديل العمليات

##### 3.16.1.3 صفحة إجراءات المستخدم (/admin/users/{id}/actions)
- Header: إجراءات المستخدم، اسم المستخدم، زر العودة
- الإجراءات مقسمة إلى مجموعات:

**Subscription:**
- تجديد الاشتراك
- تمديد الاشتراك
- إلغاء الاشتراك
- تغيير الكود

**Account:**
- تعليق الحساب
- إعادة تفعيل الحساب
- حظر الحساب
- رفع الحظر

**Security:**
- Logout All Devices
- Reset Session
- Reset Device
- Reset Limits
- **Reset Security Token**
- **Revoke Device Binding**
- **Force Re-verification**

**Operations:**
- تعديل العمليات
- تصفير العمليات
- تعديل الحد اليومي

**Logs:**
- سجل العمليات
- سجل التعديلات
- سجل تسجيل الدخول
- سجل الأخطاء
- **سجل الأمان**

- كل إجراء يحتوي على: وصف الإجراء، زر التنفيذ، Confirmation Dialog
- تسجيل كل إجراء في سجل التعديلات

##### 3.16.1.4 صفحة إدارة الاشتراك (/admin/users/{id}/subscription)
- Header: إدارة الاشتراك، اسم المستخدم، زر العودة
- عرض تفاصيل الاشتراك الحالي
- نموذج تجديد الاشتراك: اختيار مدة الاشتراك، اختيار كود جديد (اختياري)، زر تجديد
- نموذج تمديد الاشتراك: إدخال عدد الأيام، زر تمديد
- نموذج إلغاء الاشتراك: سبب الإلغاء، Confirmation، زر إلغاء
- سجل جميع التجديدات والتمديدات السابقة

##### 3.16.1.5 صفحة تعديل العمليات (/admin/users/{id}/operations)
- Header: تعديل العمليات، اسم المستخدم، زر العودة
- عرض قائمة جميع عمليات المستخدم
- لكل عملية: رقم العملية، رقم الهاتف، الفئة، المبلغ، الحالة، التاريخ، زر تعديل
- نموذج تعديل العملية: تعديل الحالة، تعديل المبلغ، إضافة ملاحظة، زر حفظ
- زر تصفير جميع العمليات (مع Confirmation)
- تسجيل كل تعديل في سجل التعديلات

#### 3.16.2 Subscriptions
- عرض قائمة الاشتراكات
- حالة كل اشتراك (نشط، منتهي، معلق)
- تاريخ التفعيل والانتهاء
- تجديد أو تعديل اشتراك

#### 3.16.3 License Keys
- إنشاء كود تفعيل جديد
- تحديد مدة الاشتراك للكود
- عرض قائمة الأكواد
- حالة كل كود (نشط، مستخدم، معطل)
- عرض المستخدم المرتبط بالكود
- تعطيل كود
- حذف كود

#### 3.16.4 Product Config

##### الوظيفة
- إدارة إعدادات جميع الكروت من جدول product_config في Supabase
- عرض قائمة جميع الكروت
- لكل كارت:
  - تفعيل/إيقاف (ON/OFF)
  - إظهار/إخفاء
  - الحالة: active، maintenance، development، unavailable، disabled_execution
  - السعر
  - الوحدات
  - مدة الصلاحية
  - الربح
  - الترتيب

## 4. القواعد والمنطق

### 4.1 نظام الصلاحيات
- User: صلاحيات المستخدم العادي (الشبكات، المفضلة، السجل، الإحصائيات، الإعدادات)
- Admin: صلاحيات إدارية (إدارة المستخدمين، الاشتراكات، الأكواد، الكروت، العمليات، الأصول المرئية، الإصدارات، الميزات، المراقبة، تقييمات الكروت، **الأمان**)
- Super Admin: صلاحيات كاملة على جميع أقسام النظام
- **التحقق من الصلاحيات يتم من السيرفر فقط، لا يتم الاعتماد على بيانات العميل**

### 4.2 منطق التفعيل
- عند إدخال كود تفعيل صحيح: تفعيل الاشتراك، تسجيل تاريخ التفعيل، حساب تاريخ الانتهاء بناءً على مدة الكود، ربط الكود بالمستخدم، تحديث حالة الكود إلى مستخدم، **ربط الكود بالجهاز (Device Binding)**
- عند إدخال كود خاطئ أو مستخدم مسبقاً: عرض رسالة خطأ، عدم السماح بالمتابعة، **تسجيل المحاولة في سجل الأمان**

### 4.3 منطق الشحن
- التحقق من صلاحية الاشتراك قبل السماح بالشحن
- التحقق من صحة رقم الهاتف والرقم السري
- التحقق من اختيار الكارت
- التحقق من حالة الكارت من product_config
- **التحقق من صحة الجلسة والجهاز قبل السماح بالشحن**
- **طلب Nonce من السيرفر قبل كل عملية شحن**
- **التحقق من صحة Nonce على السيرفر (صالح لثوانٍ محدودة)**
- إذا كان الكارت (maintenance/development/unavailable): لا يُرسل أي API، يظهر Dialog: تم إيقاف هذا المنتج مؤقتاً، وسيعود للعمل فور توفره.
- إذا كان disabled_execution: الكارت ظاهر لكن التنفيذ معطّل بنفس الـ Dialog
- عرض رسالة تنبيه قبل التنفيذ: يرجى التأكد قبل التنفيذ من عدم وجود أي مبلغ مستحق أو استلاف أو نوتة على الخط المرسل إليه، لأن ذلك قد يؤدي إلى عدم إتمام العملية بنجاح.
- منع إرسال الطلب مرتين (Request Deduplication)
- **توقيع الطلب بمفتاح يصدره السيرفر (Request Signing)**
- إرسال طلب الشحن (سيتم ربطه لاحقاً مع Recharge Engine)
- **تنفيذ عملية الشحن من السيرفر فقط (Secure Flex Execution)**
- انتظار النتيجة النهائية قبل عرض الرسالة
- تسجيل العملية في السجل
- عرض نتيجة العملية (نجاح أو فشل)
- إذا نجحت العملية: لا تظهر رسالة فشل أبداً، عرض زر تقييم الكارت (إذا كانت الميزة مفعلة)
- نظام الشحن الحالي يعمل بشكل صحيح ولا يجب تعديله

### 4.4 منطق الاشتراكات
- حساب الأيام المتبقية تلقائياً
- عند انتهاء الاشتراك: تحديث حالة الاشتراك إلى منتهي، منع المستخدم من الشحن، عرض شاشة التفعيل، إرسال إشعار تجديد

### 4.5 منطق الأكواد
- كل كود يستخدم مرة واحدة فقط
- الكود المعطل لا يمكن استخدامه
- الكود المحذوف لا يظهر في النظام

### 4.6 منطق البيانات الديناميكية
- جميع الكروت يتم تحميلها من product_config في Supabase
- جميع إعدادات الكروت ديناميكية
- لا توجد بيانات ثابتة في الكود
- النظام يعرض فقط ما يتم تحميله
- في حالة عدم توفر بيانات: عرض رسالة لا توجد كروت متاحة حالياً
- **جميع الأسعار والوحدات والصلاحيات يتم التحقق منها على السيرفر، لا يتم الاعتماد على بيانات العميل**

### 4.7 منطق الإحصائيات
- حساب جميع الإحصائيات ديناميكياً من قاعدة البيانات
- تحديث الإحصائيات تلقائياً عند إضافة عملية جديدة
- عرض Charts ديناميكية بناءً على البيانات الفعلية

### 4.8 منطق إخفاء أدوات المطورين
- إخفاء جميع أدوات المطورين من واجهة المستخدم العادي
- لا يتم حذف الكود أو إزالة الوظائف
- يتم إخفاء العناصر من واجهة المستخدم فقط
- تستمر جميع الوظائف في العمل داخلياً
- المسؤولون (Admin/Super Admin) يمكنهم الوصول لهذه الأدوات عند الحاجة

### 4.9 منطق الأصول المرئية الديناميكية
- مواقع الأصول المرئية: شعار الهيدر في أعلى اليمين، أيقونة كارت الترحيب، شعار شاشة البداية، شعارات الشبكات في صفحة الشبكات وصفحات الشبكات الفردية
- عملية التحديث: Admin يرفع صورة جديدة، رفع الملف إلى Supabase Storage، تحديث قاعدة البيانات بالرابط الجديد، تحديث فوري لجميع المستخدمين، لا حاجة لتحديث APK
- التحقق من الصيغة والحجم: التحقق من صيغة الملف (PNG, JPG, JPEG, WEBP, SVG)، التحقق من حجم الملف (أقل من 10 MB)، رفض الملفات غير المطابقة

### 4.10 منطق صفحة الشبكات
- عرض 4 بطاقات للشبكات بتصميم Premium
- كل بطاقة تعرض شعار الشبكة الديناميكي من Asset Management
- تطبيق لون الشبكة الرسمي في التصميم
- عرض عدد الخدمات (حالياً 0 لجميع الشبكات)
- عند الضغط على أي بطاقة: الانتقال إلى صفحة الشبكة المحددة
- Animations ناعمة بدون تهنيج
- بدون إعادة تحميل الصفحة

### 4.11 منطق صفحات الشبكات الفردية
- البنية الحالية: عرض Header بشعار الشبكة واسمها ووصف احترافي، عرض قسم الخدمات برسالة تحت التطوير، زر العودة للشبكات
- البنية المستقبلية (Modular): جاهزة لإضافة أقسام (فكة، دقائق، ميجابايت، باقات، كروت، عروض، خدمات)

### 4.12 منطق Offline-First
- تخزين البيانات الثابتة محلياً باستخدام Capacitor Preferences: إعدادات التطبيق، الكروت، الصور، الأيقونات
- فتح التطبيق من التخزين المحلي أولاً
- مزامنة ذكية في الخلفية
- لا skeleton طويل، لا صور مكسورة، لا تحميل بطيء

### 4.13 منطق منع التكرار
- Request Deduplication: نفس الطلب لا يُرسل مرتين
- Notification Deduplication: نفس الإشعار لا يظهر مرتين
- Toast Deduplication: نفس الرسالة لا تظهر مرتين
- Order Deduplication: نفس العملية لا تُنفَّذ مرتين

### 4.14 منطق Network Manager
- مراقبة الاتصال وجودة الشبكة
- كشف Offline واستعادة تلقائية
- إعادة مزامنة عند عودة الإنترنت
- تحسين جميع الطلبات للعمل بأقل سرعة ممكنة

### 4.15 منطق حذف المستخدم نهائياً
- حذف من: Auth، Database، Profiles، Sessions، Notifications، Activity، Logs
- إذا كان المستخدم داخل التطبيق: عرض رسالة تم حذف حسابك بواسطة الإدارة. ثم تسجيل خروج فوري

### 4.16 منطق نظام الحظر
- المحظور لا يستطيع تسجيل دخول ولا إنشاء Session
- عرض رسالة: تم حظر حسابك بواسطة الإدارة.

- كل مستخدم جديد يبدأ بإشعاراته الخاصة فقط

### 4.24 منطق تقييم الكروت واقتراح التعديلات

#### 4.24.1 عرض زر التقييم

  - بعد نجاح عملية الشحن مباشرة
  - بجانب كل عملية ناجحة في سجل العمليات

#### 4.24.2 نافذة التقييم
- عند الضغط على زر التقييم: فتح نافذة منبثقة (Popup)
- عرض معلومات العملية: رقم العملية، نوع الكارت، التاريخ

  - السعر المخصوم الفعلي من فودافون كاش (حقل رقمي، اختياري)
  - أيام الصلاحية الفعلية (حقل رقمي، اختياري)

  - اختياري
  - يسمح برفع صورة واحدة فقط
  - الصيغ المدعومة: PNG, JPG, JPEG
  - الحد الأقصى لحجم الملف: 5 MB
  - عرض الصورة بشكل مصغر بعد الرفع

- إغلاق النافذة

### 4.25 منطق الأمان (Zero Trust Architecture) - جديد

#### 4.25.1 الطبقة 1: Server Authoritative
- جميع القرارات الأمنية تتم على السيرفر
- لا يتم الاعتماد على أي بيانات من العميل
- التحقق من الصلاحيات على السيرفر فقط
- التحقق من الأسعار والوحدات على السيرفر فقط
- التحقق من حالة الاشتراك على السيرفر فقط

#### 4.25.2 الطبقة 2: APK Signature Verification
- عند فتح التطبيق: إرسال APK Signature و SHA256 و Package Name إلى السيرفر
- السيرفر يتحقق من مطابقة التوقيع مع التوقيع المعتمد المخزن في قاعدة البيانات
- إذا لم يتطابق التوقيع: رفض الاتصال، عرض رسالة: هذا الإصدار غير معتمد، يرجى تحميل التطبيق من المصدر الرسمي.
- تسجيل المحاولة في سجل الأمان

#### 4.25.3 الطبقة 3: Build Fingerprint
- عند فتح التطبيق: إرسال Build Hash و Source Hash إلى السيرفر
- السيرفر يتحقق من مطابقة الـ Hash مع الـ Hash المعتمد
- إذا لم يتطابق: رفض الاتصال، تسجيل المحاولة

#### 4.25.4 الطبقة 4: Device Binding
- عند أول تسجيل دخول: ربط الحساب بـ Android ID و Hardware Profile
- إنشاء Security Token فريد للجهاز
- تخزين Device ID و Hardware Profile و Security Token في قاعدة البيانات
- عند كل اتصال: التحقق من مطابقة Device ID و Hardware Profile و Security Token
- إذا لم يتطابق: رفض الاتصال، عرض رسالة: تم اكتشاف محاولة دخول من جهاز غير معتمد.
- السماح بربط جهاز واحد فقط لكل حساب
- Admin يمكنه إعادة ربط الجهاز (Revoke Device Binding)

#### 4.25.5 الطبقة 5: Session Signature
- عند تسجيل الدخول: السيرفر ينشئ Session مرتبطة بـ User ID و Device ID و Version
- Session تحتوي على: Session ID، Session Token، Expiry Time
- عند كل طلب: إرسال Session ID و Session Token
- السيرفر يتحقق من صحة Session
- إذا انتهت صلاحية Session: طلب تسجيل دخول جديد
- إذا تم اكتشاف Session مزورة: حظر الجهاز والمستخدم

#### 4.25.6 الطبقة 6: Request Signing
- السيرفر يصدر مفتاح توقيع (Signing Key) لكل جلسة
- عند كل طلب: توقيع الطلب بالمفتاح
- السيرفر يتحقق من صحة التوقيع
- إذا لم يتطابق التوقيع: رفض الطلب، تسجيل المحاولة

#### 4.25.7 الطبقة 7: Nonce
- عند كل عملية شحن أو Flex: طلب Nonce من السيرفر
- Nonce صالح لثوانٍ محدودة (مثلاً 30 ثانية)
- إرسال Nonce مع طلب الشحن
- السيرفر يتحقق من صحة Nonce وأنه لم يُستخدم من قبل
- إذا انتهت صلاحية Nonce أو تم استخدامه: رفض الطلب
- منع إعادة استخدام نفس Nonce

#### 4.25.8 الطبقة 8: Play Integrity
- عند فتح التطبيق: طلب Play Integrity Token
- إرسال Token إلى السيرفر
- السيرفر يتحقق من نزاهة الجهاز والتطبيق
- إذا فشل التحقق: رفض الاتصال، عرض رسالة: تم اكتشاف محاولة تشغيل التطبيق على جهاز غير آمن.
- تسجيل المحاولة في سجل الأمان

#### 4.25.9 الطبقة 9: Remote Config
- السيرفر يمكنه إيقاف أي إصدار أو جهاز عن بُعد
- Admin يمكنه حظر إصدار معين من Application Version Center
- Admin يمكنه حظر جهاز معين من Security Dashboard
- عند محاولة الاتصال: السيرفر يتحقق من حالة الإصدار والجهاز
- إذا كان محظوراً: رفض الاتصال، عرض رسالة مناسبة

#### 4.25.10 الطبقة 10: Admin Validation
- عدم الاعتماد على حالة Admin من التطبيق
- التحقق من صلاحيات Admin على السيرفر فقط
- عند كل طلب Admin: التحقق من User ID و Role من قاعدة البيانات
- إذا لم يكن Admin: رفض الطلب

#### 4.25.11 الطبقة 11: Anti Patch
- كشف Frida: فحص وجود Frida Server
- كشف Root: فحص وجود Root Access
- كشف Magisk: فحص وجود Magisk
- كشف Xposed: فحص وجود Xposed Framework
- إذا تم اكتشاف أي منها: رفض الاتصال، عرض رسالة: تم اكتشاف محاولة تشغيل التطبيق على جهاز معدل.
- تسجيل المحاولة في سجل الأمان
- حظر الجهاز تلقائياً

#### 4.25.12 الطبقة 12: Server Logging
- تسجيل شامل لكل محاولة مشبوهة:
  - APK Signature Mismatch
  - Build Fingerprint Mismatch
  - Device Binding Mismatch
  - Invalid Session
  - Invalid Request Signature
  - Invalid Nonce
  - Play Integrity Failed
  - Root/Frida/Magisk/Xposed Detected
  - Request Tampering
- تسجيل: User ID، Device ID، IP Address، Timestamp، نوع المحاولة، التفاصيل
- عرض السجلات في Security Dashboard

#### 4.25.13 الطبقة 13: Auto Ban
- حظر تلقائي للأجهزة عند:
  - 3 محاولات APK Signature Mismatch
  - 3 محاولات Device Binding Mismatch
  - 5 محاولات Invalid Session
  - اكتشاف Root/Frida/Magisk/Xposed
  - اكتشاف Request Tampering
- حظر تلقائي للجلسات عند:
  - 5 محاولات Invalid Request Signature
  - 10 محاولات Invalid Nonce
- عرض رسالة: تم حظر حسابك/جهازك بسبب نشاط مشبوه.
- إرسال إشعار للمسؤولين
- تسجيل الحظر في سجل الأمان

#### 4.25.14 الطبقة 14: No Client Trust
- عدم الثقة بأي بيانات من العميل:
  - الأسعار
  - الوحدات
  - الصلاحيات
  - الأدوار
  - حالة الاشتراك
  - حالة الكروت
- جميع البيانات يتم التحقق منها على السيرفر
- جميع الحسابات تتم على السيرفر

#### 4.25.15 الطبقة 15: Secure Flex Execution
- تنفيذ عمليات Flex من السيرفر فقط
- العميل يرسل طلب الشحن مع Nonce و Request Signature
- السيرفر يتحقق من صحة الطلب
- السيرفر ينفذ عملية Flex
- السيرفر يرسل النتيجة للعميل
- العميل لا يمكنه تنفيذ Flex مباشرة

## 6. معايير القبول

1. المستخدم يفتح التطبيق ويشاهد شاشة البداية (Splash Screen) مع الشعار الديناميكي والنصوص
2. **النظام يجري التحقق الأمني الأولي (APK Signature، Device Binding، Play Integrity)**
3. **إذا فشل التحقق الأمني: رفض الاتصال وعرض رسالة مناسبة**
